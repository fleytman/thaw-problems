# Анализ: PR #691

Создано: 2026-06-10

## Короткий вывод

PR #691 выглядит как сильный и добросовестный ответ на issues. Он реализует центральный запрос из #685: just-in-time confirmation на автоматическом пути display transition, причём только в ситуации, где применение spacing действительно приведёт к relaunch wave. Он также существенно закрывает #686: новый warning text говорит про apps with menu bar items и про возможную потерю unsaved input/progress/transient state.

Если PR влить как есть, он, скорее всего, закрывает основной UX/safety gap, о котором мы писали. Но остаются два UX-риска, которые стоит либо попросить проверить, либо иметь в виду при ответе.

## Что PR исправляет хорошо

### 1. JIT confirmation на реальном display-transition relaunch

`DisplaySettingsManager.applyActiveDisplaySpacing` теперь отдельно обрабатывает автоматический путь `reason == "screenParametersChanged"` и спрашивает пользователя, только если одновременно выполняются условия:

- `confirmSpacingRelaunch == true`;
- `spacingManager.willRelaunch(forOffset: desired) == true`;
- `presentSpacingRelaunchConfirmation() == true`.

Это именно та форма условия, которую мы пытались описать: не показывать предупреждение на каждый display event, а показывать его только тогда, когда Thaw действительно собирается переписать system-wide spacing и перезапустить приложения с menu bar items.

### 2. No-op guard перед prompt

`MenuBarItemSpacingManager.willRelaunch(forOffset:)` повторяет no-op guard из `applyOffsetLocked`: он сравнивает целевые spacing/padding с текущими on-disk values. Это важно, потому что без такой проверки Thaw мог бы спрашивать пользователя даже в ситуациях, где relaunch wave фактически не будет.

### 3. Cancel/postpone behavior

Если пользователь нажимает `Cancel` в JIT alert, PR не применяет spacing, оставляет текущий on-disk spacing и специально не обновляет `lastAppliedActiveDisplayUUID`. Поэтому следующий настоящий переход дисплея сможет снова показать prompt.

Это разумная семантика “отложить / пока оставить текущий spacing”, а не скрытый irreversible action.

### 4. Warning wording ближе к тому, что мы просили

Новый persistent warning:

> When a display transition requires Thaw to apply a different menu bar spacing, Thaw relaunches apps with menu bar items. Relaunching apps may cause unsaved input, progress, or transient app state to be lost.

Это хорошо совпадает с уточнённой формулировкой: проблема не в каждом display event вообще, а в display transition, который требует применить другой spacing. И отдельно назван главный safety-impact: потеря unsaved input/progress/transient state.

### 5. Manual Apply/global warnings усилены

Warning для active-display Apply теперь говорит, что применение spacing перезапустит каждое приложение с menu bar item и может привести к потере unsaved input/progress/transient state.

Global apply wording тоже усилен: если spacing активного дисплея изменится, Thaw перезапустит apps with menu bar items, и это может привести к потере состояния.

### 6. Есть bypass для advanced users

Новый `confirmSpacingRelaunch` toggle включён по умолчанию. Если пользователь осознанно принимает риск, он может отключить confirmations. Это в целом совпадает с моделью, которую предлагал `nightah`: JIT warning + bypass для тех, кто понимает последствия.

## Риски и вопросы

### 1. Toggle `Confirm before relaunching apps` отключает не только relaunch confirmation, но и global overwrite confirmation

В `DisplaySettingsPane.requestGlobalApply` сейчас есть такой path:

```swift
if !displaySettings.confirmSpacingRelaunch {
    commitGlobalApplyWithoutConfirmation(activeProfileID: activeID)
    return
}
```

То есть если пользователь выключил `Confirm before relaunching apps`, Thaw сразу вызывает `commitGlobalApplyWithoutConfirmation(activeProfileID:)`.

Проблема в том, что `Apply to All Displays` - это не только подтверждение relaunch. Это ещё и операция broadcast/global overwrite: `commitGlobalApply()` вызывает `displaySettings.applyGlobalToAllKnownDisplays()`, то есть применяет global template ко всем известным дисплеям.

Из-за этого scope toggle становится шире, чем видно из UI:

- пользователь может выключить `Confirm before relaunching apps`, потому что он понял и принял только relaunch-risk;
- позже он нажимает `Apply to All Displays`;
- Thaw без отдельного подтверждения перезаписывает configurations всех известных дисплеев global template;
- пользователь мог не понимать, что выключение relaunch-confirmation также выключило safety prompt для массового overwrite.

Почему это важно: global overwrite - отдельная destructive/broad operation. Даже если пользователь согласен не подтверждать каждый future relaunch, это не обязательно означает согласие отключить confirmation перед перезаписью настроек всех дисплеев.

Более безопасные варианты:

- оставить global overwrite confirmation всегда, даже если relaunch confirmations выключены;
- разделить две настройки: `Confirm before app relaunches` и `Confirm before applying global template to all displays`;
- либо явно написать в toggle annotation, что отключение этого toggle also disables confirmation before broadcasting the global template to all displays.

### 2. `Don't ask again` в JIT alert слишком расплывчат для destructive bypass

В `presentSpacingRelaunchConfirmation()` alert показывает suppression checkbox с текстом:

> Don't ask again

После `runModal()` код делает:

```swift
if alert.suppressionButton?.state == .on {
    confirmSpacingRelaunch = false
}
return response == .alertFirstButtonReturn
```

То есть если checkbox включён, будущие confirmations отключаются независимо от того, нажал пользователь `Apply` или `Cancel`.

Это может быть неожиданным:

- пользователь может понимать `Don't ask again` как “не спрашивать больше про это конкретное событие”;
- или как “я сейчас отменяю, и не хочу видеть этот prompt в этом конкретном сценарии”;
- но реальный эффект - future display transitions будут применять spacing и relaunch apps без confirmation.

Особенно рискован край `Don't ask again` + `Cancel`: пользователь отказывается от текущего relaunch, но одновременно отключает будущие предупреждения. Потом следующий display transition может снова привести к silent apply/relaunch, хотя пользователь мог не иметь в виду такой глобальный bypass.

Более безопасные варианты:

- заменить label на более явный: `Don't ask again and apply future spacing changes without confirmation`;
- honour suppression checkbox только если пользователь нажал `Apply`;
- если пользователь нажал `Cancel` с включённым suppression, не выключать future confirmations или показывать отдельное подтверждение такого bypass.

### 3. Minor wording nit: `each app with a menu bar item`

Фраза `each app with a menu bar item` в целом понятна, но чуть яснее было бы:

> each app that has a menu bar item

Это не принципиальный blocker. Просто формулировка меньше оставляет места для неверного чтения.

## Общая позиция

PR в целом существенно закрывает наши основные concerns. Правильный ответ на него должен быть позитивным и коротким, без повторного спора с maintainer.

Предлагаемый stance:

- поблагодарить `nightah` за реальный JIT path;
- сказать, что это совпадает с уточнённой проблемой: prompt появляется только когда display transition действительно потребует применить другой spacing и может вызвать relaunch wave;
- попросить double-check по одному важному UX-краю: отключение `Confirm before relaunching apps` сейчас, похоже, также отключает global overwrite confirmation, хотя это более широкое действие;
- опционально упомянуть `Don't ask again`, потому что сейчас checkbox отключает future confirmations даже после `Cancel`.
